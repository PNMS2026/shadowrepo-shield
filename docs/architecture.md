# Architecture Documentation

This document describes the design, data flows, and technical implementation details of **ShadowRepo Shield**.

---

## 1. System Context

ShadowRepo Shield runs entirely as a desktop application. It integrates a React frontend running inside a webview with a Rust core backend that executes file operations and static analysis.

```
+-------------------------------------------------------------+
|                       Tauri v2 Desktop                      |
|                                                             |
|   +-----------------------+         +-------------------+   |
|   |   React + TypeScript  |  IPC    |    Rust Backend   |   |
|   |       Frontend        |<------->|    Core Engine    |   |
|   +-----------------------+         +-------------------+   |
|               |                              |              |
+---------------|------------------------------|--------------+
                v                              v
      +-------------------+          +-------------------+
      |  Ethers.js / RPC  |          | SQLite Database   |
      |   (Smart Contract)|          | (Local Reports)   |
      +-------------------+          +-------------------+
```

## 2. Component Design & Implementation

### 2.1 Static Directory Walker (`walker.rs`)
The walker module is responsible for traversing the targeted repository path.
- **Constraints & Pruning:** To maximize scan efficiency and prevent memory fatigue, it prunes known dependency trees (`node_modules`, `bower_components`, `.git`, `build`, `dist`, `out`, `.next`, `.svelte-kit`).
- **Binary Filtering:** To prevent buffer overflow risks and unreadable parses, it automatically excludes binary extensions (e.g., `.png`, `.jpg`, `.woff`, `.wasm`, `.zip`, `.exe`, `.dll`) and files larger than 10MB.
- **Symbolic Links:** Ignores symbolic links to prevent circular references or path traversal outside the scan scope.

### 2.2 Threat patterns engine (`patterns.rs`)
The analysis engine tokenizes the source code files line-by-line and applies highly optimized regular expression patterns. Threats are split into four main categories:
1. **Lifecycle Scripts:** Scanning `package.json` for suspicious trigger actions (`preinstall`, `postinstall`, `install`, `prepare`) which could execute untrusted commands upon installation.
2. **Suspicious Code:** Finding dangerous Node.js or JavaScript APIs (`child_process`, `exec()`, `spawn()`, `eval()`, `new Function()`, `readFileSync(".env")`, private key or seed phrase accesses).
3. **Wallet Drainers:** Recognizing client-side Web3 malicious practices (`setApprovalForAll`, `MaxUint256` approvals, permit signature abuses, batch ERC-20 `transferFrom` loops, mnemonics key extraction).
4. **Solidity Risks:** Inspecting Smart Contracts for structural hazards (`tx.origin` authentication, `delegatecall` injection vectors, `selfdestruct` loops, unchecked low-level calls, centralized minting/blacklisting controls).

### 2.3 Risk Calculator (`risk.rs`)
The hazard index is calculated using a weighted risk formula:
- **Base Score:** Calculated by adding findings weighted by severity:
  - `Critical` finding: 30 points
  - `High` finding: 15 points
  - `Medium` finding: 5 points
  - `Low` finding: 1 point
- **Modifier Multipliers:**
  - Active `package.json` pre/postinstall hook presence adds a **+10 point** penalty.
  - Hardcoded sensitive file matches (like a `.env` or raw private key) adds a **+15 point** penalty.
- **Limits:** The score is capped at `100`.
- **Classification Mapping:**
  - `0 - 20`: Low Risk
  - `21 - 50`: Medium Risk
  - `51 - 80`: High Risk
  - `81 - 100`: Critical Risk

### 2.4 Local SQLite Database (`db.rs`)
A local-first SQLite file (`shadowrepo_shield.db`) is stored inside the app's sandboxed data directory.
- **`scans`:** Stores scanning metadata, risk score, repository path, hashes, and on-chain TX status.
- **`findings`:** Stores granular vulnerability details. Has a foreign key reference to `scans` with a `ON DELETE CASCADE` constraint. When a scan is deleted, its findings are cleared automatically.
- **`settings`:** Stores single-row user settings (Storage export paths, RPC network, smart contract address, etc.).

### 2.5 Cryptographic Hasher (`hasher.rs`)
To ensure trustless verification without exposing private source code, the system generates two hashes:
- **Repository Hash:** Generated deterministically by walking through the repository file-by-file in alphabetical order, hashing the relative path and contents of each file with SHA-256, and combining the hashes.
- **Report Hash:** Generated by taking the JSON representation of the scan result structure and running it through SHA-256.

### 2.6 Smart Contract `ShadowRepoProof.sol` (Deferred / Demo-Only)
Deployed on EVM-compatible networks, this contract registers security check proofs.
- **Verification Trust Boundary:** In v1.4.0, direct blockchain submission of "verified" flags from the user client is disabled because user submissions are untrusted. Future verified on-chain proofs require a `trustedVerifier` wallet signature matching the official CI pipeline. The current implementation is retained for demonstration purposes only.
- **Proofs Storage:** Indexed by `reportHash` to allow external validation of a scan's score, mode, and metadata.

### 2.7 Command Line Interface (CLI) (`cli.rs`)
A standalone binary `shadowrepo-cli` is compiled for native execution (e.g., in CI pipelines or terminal environments).
- **Execution Mode:** Can run scans in either `--mode local` or `--mode verified`. Verified mode is only trusted when executed by the official GitHub Action using the official release binary and checksum verification. A locally executed `--mode verified` scan must not be treated as public proof.
- **CI Attestation:** When run inside official GitHub Actions, the CLI reads environment variables (such as `GITHUB_RUN_ID` and `GITHUB_SHA`) to build a deterministic `scanner_signature` representing the verified build environment.
- **Exit Codes:** Maps scan risk levels to exit codes: `0` (Low/Review), `2` (High Risk), and `3` (Critical Threat Indicators).

### 2.8 AI Security Advisory (`ai.rs`)
Integrates OpenAI-compatible, Gemini, NVIDIA NIM, and OpenRouter models to offer security guidance.
- **Security Constraints:** The AI behaves strictly as an advisor. It receives read-only finding metadata (descriptions, file paths, and matched pattern strings) and is physically blocked from modifying the deterministic scan score, severity level, or risk level.
- **Privacy Controls:** To protect proprietary code, full source code files are never uploaded. Furthermore, sensitive strings (like private keys, API secrets, and mnemonics) are redacted locally via regex before sending prompt contents to LLM endpoints.

## 3. Core Data Flow

### 3.1 Scanning Flow (Local Desktop)
1. **User Action:** User inputs a local path or uploads a ZIP file.
2. **Backend Unzipping (if ZIP):** Extracts ZIP contents to a secure temp folder.
3. **Traversing:** `walker.rs` filters out binary files and prunes heavy directories.
4. **Parsing & Hashing:** `patterns.rs` runs regular expression checks; `hasher.rs` computes repository and report SHA-256 hashes.
5. **Persisting:** `db.rs` writes the scan to the local SQLite database.
6. **UI Update:** The React page displays findings alongside a "🟡 Local Scan" badge.

### 3.2 Proof Submission Flow (Demo-Only)
1. **User Request:** User clicks "Anchor Proof on Blockchain" on the Scan Result page.
2. **On-chain Anchor:** Calls `submitProof()` on the smart contract.
3. **Transaction Update:** Upon receipt, Tauri updates the SQLite record with the tx hash.

### 3.3 CI Verified Scanning Flow (GitHub Actions)
1. **Action Trigger:** A push or pull request triggers the official ShadowRepo Shield Action.
2. **Official Binary Download:** The Action downloads the official release binary matching the repository owner's release. It verifies the binary's SHA-256 checksum against the official release checksum.
3. **Scan Execution:** The binary scans the checked-out codebase in `--mode verified`. It computes the report hash and compiles the `scanner_signature` from the GITHUB environment variables.
4. **Artifact Upload:** The resulting `report.json` is uploaded as a workflow artifact, and a PR comment is created summarizing the metrics.

### 3.4 AI Advisory Flow
1. **Trigger:** The user requests AI explanation for a single finding, or requests a complete posture advisory.
2. **Redaction:** Relative file paths and matched code snippets are scrubbed of credentials/private keys using `redact_secrets()`.
3. **LLM Query:** The redacted metadata is bundled into a standardized prompt and dispatched to the user-selected LLM provider.
4. **Rendering:** The frontend displays the AI advisory summary, actionable recommendations, and confidence score, clearly labeled as advisory-only.

## 4. Trust & Security Model

```
┌────────────────────────────────────────────────────────┐
│                     TRUST BOUNDARY                     │
├────────────────────────────────────────────────────────┤
│                                                        │
│   [Local Scan] ──> Self-Reported ──> Low Trust (Badge)  │
│                                                        │
│   [Verified Scan] ──> Run in CI ──> Higher Trust       │
│      └─ Verified via Official Download Checksum        │
│      └─ Signed with Environment-Derived Signature      │
│                                                        │
│   [Report Integrity]                                   │
│      └─ SHA-256 hash covers ALL scan metrics           │
│      └─ CLI / Desktop verifies hash matches database   │
│                                                        │
└────────────────────────────────────────────────────────┘
```
- **Dual Scan Badges:** Clearly separates scans into "Local Scan" (unverified self-check) and "CI Verified Foundation" (scanned by official CI runner) to prevent social engineering.
- **Report Integrity Hashing:** Re-serializing the scan result struct (excluding the `report_hash` field itself) and hashing it protects findings, scores, and scan modes from external JSON tampering.
